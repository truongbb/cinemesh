# Brainstorm: Request ID Tracing

### Goal
Ensure a single, consistent Request ID (UUID without dashes) is generated at the Gateway, propagated through all downstream microservices (Service A -> B -> C), printed in all service logs via MDC, and returned to the frontend in the final HTTP response.

### Constraints
- Must use `X-Request-Id` header (or similar standard).
- Must use MDC to ensure logs capture the trace ID properly.
- Must work across Spring Cloud Gateway (Reactive/WebFlux) and Spring Boot services (Servlet + OpenFeign).
- Must resolve the current issue where every service generates its own ID.

### Known context
- Currently, Gateway generates an ID in `CorrelationIdGlobalFilter` and mutates the request header.
- Downstream services use `MdcLogFilter` to extract `X-Request-Id` or generate a new one if missing, storing it in MDC.
- Feign uses two overlapping interceptors (`FeignCorrelationInterceptor` and `FeignClientConfig`) to pass the ID to the next service.
- The user observes that every service currently generates a separate request ID.

### Risks
- **Thread Context Loss**: Feign calls or async processes might run on different threads, causing MDC and RequestContextHolder to be empty when Feign interceptors run, dropping the header.
- **Interceptor Conflicts**: Multiple Feign interceptors are currently registered, which can lead to unpredictable header behavior or duplication.
- **Response Mutation**: Gateway currently mutates the request but not the response, meaning the frontend might receive a different ID (generated by Service A) instead of the Gateway's ID.

### Options (2–4)

**Option 1: Fix Custom Filters & Consolidate Feign Interceptors**
- **Summary**: Fix the Gateway `CorrelationIdGlobalFilter` to attach the ID to both Request and Response. Consolidate `FeignClientConfig` and `FeignCorrelationInterceptor` into a single, reliable interceptor. Ensure thread propagation is handled if Feign uses different threads.
- **Pros / cons**: Keeps the existing custom logic with minimal code changes. Con: Doesn't natively solve async thread context loss if services start heavily using `@Async` or Resilience4j thread pools in the future.
- **Complexity / risk**: Low.

**Option 2: Adopt Micrometer Tracing (Spring Boot 3 Standard)**
- **Summary**: Replace custom MDC filters and Feign interceptors with Spring Boot 3's built-in observability features (`micrometer-tracing-bridge-brave`). Let the framework handle propagating `traceId` (Request ID) across WebFlux, Servlets, and Feign automatically, including thread-hopping scenarios.
- **Pros / cons**: Extremely robust, native support for log correlation, easily integrates with ELK (via Logstash or OTLP). Con: Requires adding new dependencies and removing the current custom interceptor code.
- **Complexity / risk**: Medium (due to dependency changes), but lowest long-term maintenance risk.

**Option 3: Use strict ThreadLocal Propagation for Feign**
- **Summary**: Implement a custom `Callable` or `Executor` wrapper that copies the MDC context to any new threads spawned by Feign, ensuring the Request ID is never lost.
- **Pros / cons**: Solves the root cause if thread-hopping is the issue. Con: Over-engineering if standard Spring properties can solve it.
- **Complexity / risk**: Medium.

### Recommendation
**Recommendation: Option 1** is the fastest path to fix your current implementation. We will:
1. Fix Gateway's `CorrelationIdGlobalFilter` so it adds the Request ID to the Response back to the FE.
2. Remove the redundant Feign interceptor code from `FeignClientConfig` to prevent conflicts with `FeignCorrelationInterceptor`.
3. Verify that `MdcLogFilter` successfully extracts the Gateway ID and that Feign passes it correctly without thread context loss.

*(Note: If you plan to scale this extensively, Option 2 is highly recommended as the industry standard).*

### Acceptance criteria
- Gateway generates a single Request ID and adds it to the downstream request AND the frontend response.
- Service A receives the ID, logs it, and passes it to Service B via Feign.
- Service B receives the same ID, logs it, and passes it to Service C.
- The FE receives the exact same Gateway-generated ID.
- No duplicate request IDs are generated per single request chain.
